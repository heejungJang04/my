[화면을 만드는 손과 데이터를 읽는 눈을 가진 개발자]

저는 프론트엔드 개발을 디자인을 화면으로 옮기는 일이 아니라, 사용자의 행동과 서버의 데이터가 만나는 접점을 설계하는 일이라고 생각합니다. 버튼 하나에도 로딩·에러·중복 요청 같은 상태가 얽혀 있고, 이를 어떻게 다루느냐가 서비스의 완성도와 신뢰를 결정하기 때문입니다.

제가 React와 TypeScript를 선택한 이유도 여기에 있습니다. React는 화면을 상태(state) 단위로 쪼개어 복잡한 인터랙션을 예측 가능하게 관리하고, TypeScript는 서버와 주고받는 데이터 구조를 타입으로 고정해 협업 중 생기는 오류를 미리 잡아줍니다. 상태와 타입을 안정적으로 통제할 수 있다는 점이 곧 신뢰할 수 있는 화면의 토대가 된다고 보았습니다.

이 관점은 직접 서비스를 만들고 배포해 운영하며 더 단단해졌습니다. 제가 만든 화면이 실제 사용자에게 닿는 과정을 겪으며 '동작하는 화면'과 '신뢰할 수 있는 화면'은 다르다는 것을 체감했고, 그 차이는 중복 요청을 막고 데이터 정합성을 지키며 사용자가 당황할 상황을 미리 없애는 작은 디테일에서 나온다는 것을 배웠습니다. 동시에 공공 데이터를 분석하며, 화면 너머의 데이터가 곧 다음 기능의 근거가 됨도 알게 되었습니다.

이렇게 만드는 일과 데이터를 다루는 일을 함께 해오며, 두 영역을 잇는 것이 제 강점이 되었다고 느낍니다. 직접 만들어봤기에 데이터를 어떤 화면으로 풀어낼지 고민할 수 있었고, 데이터를 다뤄봤기에 어떤 기능이 필요할지 함께 생각할 수 있었습니다. 이런 경험을 살려 사용자에게 더 정확하고 믿을 수 있는 서비스를 만드는 데 보탬이 되고 싶어 이 직무에 지원했습니다.


[문항 2] 지원 직무와 관련한 본인의 역량 또는 경험에 대해
         구체적으로 작성해주세요.  (약 900자)
───────────────────────────────────────────────────────────

저는 사용자에게 닿는 화면을 만드는 일과, 그 바탕이 되는 데이터를 다루는 일을 모두 경험했습니다. AI·데이터 동아리의 운영진으로 자체 플랫폼을 직접 개발했고, 대표적으로 게시판 댓글 기능에서 TypeScript로 API 응답 구조를 interface로 고정해 협업 중 혼선을 줄였습니다. 비동기 처리에서 버튼을 다시 눌렀을 때 중복 댓글이 생기는 문제를 isSubmitting 플래그로 막았고, 댓글 수는 서버 재조회와 낙관적 업데이트로 정확성과 응답성을 함께 잡았으며, 입력은 dompurify로 sanitize해 XSS를 차단했습니다. 나아가 기능을 넘어 사용자 경험의 빈틈까지 챙겨, 댓글 삭제·작성 시 페이지 경계를 자동 보정해 빈 화면이나 누락 없이 결과가 곧바로 보이도록 했고, 입력·처리 상태에 따라 버튼 노출과 피드백을 즉각 반영했습니다.

데이터를 직접 만들어 다루는 일에도 적극적입니다. 딥페이크 탐지용 데이터셋(ReGen-DF)을 구축하며 여러 생성형 AI를 직접 유료로 활용해 위조 얼굴을 만들고 GCP 클라우드에서 실행했고, 동아리 데이터 경진대회를 직접 기획해 12만여 개 게임 데이터에서 정답을 누설하는 누수 컬럼을 제거하고 식별자를 익명화한 train/test 데이터셋을 설계하며 데이터 무결성까지 다뤘습니다.

이렇게 저는 화면부터 데이터까지 직접 만들고 다루며 해결해 왔습니다.


[문항 3] 사업 참여기간 동안 이루고 싶은 목표와
         향후 발전계획에 대해 작성해주세요.  (약 800자)
───────────────────────────────────────────────────────────

지금까지 저는 React·TypeScript로 웹 서비스를 만들고 Vercel로 배포하며 프론트엔드를, 공공 데이터(HIRA) 분석으로 데이터 역량을 쌓아왔습니다. 이번 인턴십에서는 이 경험을 실무에 적용하며 현업의 개발·협업 방식을 빠르게 익히는 것이 목표입니다.

동시에 저는 빠르게 바뀌는 기술 흐름에 열려 있는 사람이 되고자 합니다. 최근 LLM을 비롯한 AI 기술이 다양한 분야에 접목되는 흐름에 관심을 갖고 공부하고 있으며, 이런 변화에 맞춰 제 역량을 꾸준히 넓히고 싶습니다. 어떤 직무를 맡든, 새로운 기술을 빠르게 학습해 실무에 녹여내는 적응력이 중요하다고 믿습니다.

특히 저는 AI를 잘 활용하는 것 자체가 중요한 역량이 되었다고 봅니다. 평소 Claude Code, GitHub Copilot 같은 AI 코딩 도구를 일상적으로 활용해, 반복적인 코드 작성 시간을 줄이고 작성한 코드를 검토·개선하며 생산성과 코드 품질을 함께 높이고 있습니다. 이러한 AI 활용 능력을 실무에서 더 전문적으로 키우고 싶습니다.

이렇게 쌓아온 개발·데이터 경험과 AI 활용 역량을 바탕으로, 실무에 빠르게 적응하고 팀에 실질적인 도움이 되는 사람으로 성장하겠습니다.

[무박 5일, 막다른 데이터 앞에서 팀의 방향을 다시 세운 경험]

5일간 진행된 무박 데이터 공모전에서 저희 팀은 위암 환자의 담석증 위험을 예측하는 모델을 만들기로 하고 역할을 나눠 분석을 시작했습니다. 그러나 데이터를 파고들수록 위암·위절제술·담석을 모두 가진 환자가 1,553명 중 단 1명뿐이라는 사실이 드러났고, SMOTE 증강으로도 모델 학습이 불가능했습니다. 마감은 다가오는데 프로젝트의 핵심이 무너졌고, 팀원들도 "이대로 괜찮을까" 막막해했습니다.

저는 이 데이터를 '실패'가 아니라 '다른 방향의 근거'로 봐야 한다고 판단했습니다. 모델은 포기하되, 분석이 타당함을 증명하면 의미 있는 서비스로 전환할 수 있다고 보았기 때문입니다. 그래서 직접 심평원 약가코드 데이터에서 주성분 코드를 찾아 데이터 속 처방을 추적했고, 마스킹된 부분은 약효 분류번호까지 역으로 확인해 그 약이 담석 완화제(이담제)임을 밝혀냈습니다. 이렇게 '우리 분석이 외부 논문·실제 처방과 일치한다'는 근거를 팀에 공유하자, 막막하던 분위기가 "이걸로 자가관리 서비스를 만들어보자"는 방향으로 환기되었습니다.

팀은 이 분석을 토대로 위험 점수·복약 알림·커뮤니티 기능을 갖춘 자가관리 앱을 기획했고, 결과적으로 수상까지 이어졌습니다. 저는 협업이 막혔을 때 필요한 것은 감정적 위로가 아니라 방향을 바꿀 수 있는 객관적 근거를 만들어 공유하는 것임을 배웠습니다.
